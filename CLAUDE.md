## 4. Requerimientos del Proyecto Final (resumen del PDF)

**Objetivo:** App móvil nativa para que la comunidad de ESCOM gestione su
calendario de Exámenes a Título de Suficiencia (ETS). Debe ser **Offline-first**,
manejar estados complejos y ofrecer UX de alto nivel con **Material Design 3**.

### Módulo Público (Consulta)
1. Buscador inteligente con filtros por Carrera, Semestre y Unidad de Aprendizaje.
2. Tabla de resultados dinámica: Materia, Fecha, Turno, Salón, Profesor evaluador.
3. Exportación del calendario a PDF (y .ics iCalendar para puntos extra).

### Módulo Administrativo (Gestión)
1. Autenticación: login seguro con contraseñas encriptadas.
2. Dashboard con estadísticas rápidas (ej. exámenes por carrera).
3. CRUD completo de la oferta de exámenes.
4. Gestión de catálogos: Carreras y Edificios/Salones.

### Requerimientos Técnicos (irrenunciables)
- **Tecnología:** Dart / Flutter (compatible con Stable 3.x).
- **Estado:** prohibido `setState()` para lógica de negocio. Usar
  obligatoriamente Provider, Riverpod o BLoC.
- **Arquitectura:** Clean Architecture — separación Data / Domain / Presentation.
- **API REST:** consumir endpoints de un backend; modelos robustos con
  serialización JSON (`fromJSON`); manejo profesional de errores de red
  (Timeout, 404, 500) con Snackbars o diálogos.
- **Persistencia:** caché local con `sqflite` o `shared_preferences` para
  exámenes favoritos/guardados.
- **Notificaciones:** alertas locales para recordar fecha de examen.
- **Interoperabilidad:** `url_launcher` para geolocalización de salones o contacto.

### Indicaciones para reducir ambigüedad
1. Flutter Stable 3.x.
2. Obligatorio `flutter_lints`; warnings de análisis estático restan puntos.
3. Assets (imágenes y fuentes) declarados en `pubspec.yaml`; nada hardcoded.
4. Se valora positivamente `get_it` (o similar) para inyección de dependencias.

## 5. Manejador de estado (DEFINIDO)

**Riverpod con sintaxis de anotaciones (`@riverpod`).**

- Razón: es el enfoque que el profesor más trabajó en clase (ver los múltiples
  PDFs de Riverpod en la sección 2, incluyendo `RiverpodConAnotaciones.pdf` y la
  versión generada 3.3.1). Encaja con Clean Architecture y maneja estados
  asíncronos de la API REST mediante `AsyncValue` (loading/error/data).
- Paquetes: `flutter_riverpod` + `riverpod_annotation` (runtime) y
  `riverpod_generator` + `build_runner` + `custom_lint` + `riverpod_lint` (dev).
- Patrón: providers organizados por capa (data / domain / presentation). Usa
  `AsyncNotifier` / `Notifier` con anotaciones; evita `setState()` para lógica
  de negocio (prohibido por el proyecto).

## 7. Comandos de terminal para instalar todo (los corre el usuario)

> El usuario ejecuta estos comandos en SU terminal. Claude Code: indica estos
> pasos y, si el proyecto ya está creado, puedes editar el `pubspec.yaml` y
> pedir que se corra `flutter pub get`.

### Verificar entorno
```bash
flutter --version        # confirmar Flutter Stable 3.x
flutter doctor           # revisar que no falte nada (Android SDK, etc.)
```

### Crear el proyecto (si aún no existe)
```bash
flutter create gestion_ets
cd gestion_ets
```

### Dependencias de runtime
```bash
flutter pub add flutter_riverpod riverpod_annotation
flutter pub add sqflite path           # persistencia local (offline-first)
flutter pub add shared_preferences     # favoritos / preferencias simples
flutter pub add http                   # consumo de API REST (o usar dio)
flutter pub add get_it                 # inyección de dependencias
flutter pub add flutter_local_notifications   # recordatorios de examen
flutter pub add url_launcher           # geolocalización de salones / contacto
flutter pub add intl                   # formateo de fechas
flutter pub add pdf printing           # exportar calendario a PDF
flutter pub add add_2_calendar         # exportar a .ics (puntos extra)
```

### Dependencias de desarrollo (codegen + lints)
```bash
flutter pub add dev:build_runner dev:riverpod_generator dev:riverpod_lint dev:custom_lint
flutter pub add dev:flutter_lints      # obligatorio por el proyecto
```

### Generar código de Riverpod (cada vez que cambien los providers anotados)
```bash
dart run build_runner build --delete-conflicting-outputs
# o en modo watch mientras desarrollas:
dart run build_runner watch --delete-conflicting-outputs
```

### Habilitar el lint de Riverpod en analysis_options.yaml
Agregar:
```yaml
analyzer:
  plugins:
    - custom_lint
```

### Verificar que todo quedó bien
```bash
flutter pub get
flutter analyze          # no debe haber warnings (restan puntos)
flutter run              # correr en emulador / dispositivo
```

> Nota: `http` vs `dio` — el proyecto pide manejo profesional de errores de red
> (Timeout, 404, 500). `dio` lo facilita con interceptores; `http` es más simple
> y suficiente. Si prefieres dio, reemplaza `flutter pub add http` por
> `flutter pub add dio`.
